# Integration Guide

Reference documentation for the dev-integrate skill.

## Merge Strategy

### Branch Ordering

The merge process uses topological sort (Kahn's algorithm) based on `depends_on`
declarations in flow.json. This ensures that:

1. Independent subtasks (no dependencies) are merged first
2. Dependent subtasks are merged only after all their prerequisites
3. The final merged state reflects the correct layering of changes

### Merge Commit Style

Each subtask is merged with `--no-ff` to preserve the branch history and create
an explicit merge commit:

```
git merge --no-ff $BRANCH -m "merge: integrate $TASK_ID ($SCOPE)"
```

This produces a clear, auditable history showing exactly when each subtask was
integrated.

### Example Merge Order

```
flow.json subtasks:
  task1: depends_on: []         # auth module
  task2: depends_on: [task1]    # user API (needs auth)
  task3: depends_on: []         # logging utility

Computed merge order: task1 -> task3 -> task2

Rationale:
  - task1 and task3 have no dependencies (merged first)
  - task2 depends on task1 (merged after task1)
  - task3 is independent of task1 (order between them is arbitrary)
```

## Conflict Resolution Patterns

### Auto-Resolvable Conflicts

The following file types are auto-resolved by accepting the incoming branch version
(`--theirs`), since they are typically regenerated:

| File Pattern | Strategy | Rationale |
|-------------|----------|-----------|
| `*.lock` | Accept theirs | Lock files are regenerated from manifests |
| `*.sum` | Accept theirs | Go module checksums are recomputed |
| `package-lock.json` | Accept theirs | Regenerated by npm install |
| `yarn.lock` | Accept theirs | Regenerated by yarn install |
| `pnpm-lock.yaml` | Accept theirs | Regenerated by pnpm install |

After auto-resolving lock file conflicts, run the appropriate package manager to
regenerate a correct lock file:

```bash
# Node.js
npm install    # or yarn install / pnpm install

# Go
go mod tidy
```

### Additional Auto-Resolution Patterns

Beyond lock files, the following patterns can be auto-resolved:

| Pattern | Strategy | Rationale |
|---------|----------|-----------|
| Import order conflicts | Combine both imports and sort | Non-semantic ordering |
| Adjacent additions | Keep both added lines | Non-overlapping additions |
| Same file, different sections | Keep both changes | Changes don't interact |

For code logic conflicts, analyze context from both branches to understand intent,
then attempt manual merge (1 attempt). If unresolvable, abort.

### Unresolvable Conflicts

Code file conflicts require manual intervention. When an unresolvable conflict is
detected:

1. The merge is aborted (`git merge --abort`)
2. The conflict details are recorded in the merge results JSON
3. The process stops and reports which files conflict
4. User must resolve conflicts manually and re-run integration

### Common Conflict Scenarios

| Scenario | Likely Cause | Resolution |
|----------|-------------|------------|
| Same function modified | Overlapping subtask scope | Manual merge, review both changes |
| Import conflicts | Multiple subtasks add imports | Combine import statements |
| Config file conflicts | Shared configuration modified | Merge config sections manually |
| Test file conflicts | Tests for same module | Combine test suites |

## Type Check Configuration

### TypeScript

```bash
# Detection
[ -f "tsconfig.json" ]

# Command
npx tsc --noEmit

# Common post-merge issues
# - Missing type imports from merged branches
# - Conflicting type declarations
# - Incompatible interface extensions
```

### Python

```bash
# Detection
[ -f "setup.py" ] || [ -f "pyproject.toml" ]

# Command
mypy .

# Alternative (if mypy not configured)
python -m py_compile *.py

# Common post-merge issues
# - Incompatible type annotations
# - Missing stub packages
# - Circular import issues
```

### Go

```bash
# Detection
[ -f "go.mod" ]

# Command
go vet ./...

# Additional checks
go build ./...

# Common post-merge issues
# - Unused imports from merged code
# - Interface implementation mismatches
# - Package naming conflicts
```

### Rust

```bash
# Detection
[ -f "Cargo.toml" ]

# Command
cargo check

# Common post-merge issues
# - Trait implementation conflicts
# - Lifetime annotation mismatches
# - Feature flag conflicts in Cargo.toml
```

## Integration Testing Approach

### Pre-Validation

Before running full integration tests, perform a quick sanity check:

1. Verify all expected files exist in the merge worktree
2. Run type checks to catch compilation errors early
3. Run fast unit tests before slower integration tests

### Validation via dev-validate

The integration uses the existing dev-validate skill:

```bash
Skill: dev-validate --worktree $MERGE_WORKTREE
```

This auto-detects the project type and runs the appropriate test suite and linter.

### Post-Merge Validation Checklist

- [ ] Type check passes (tsc/mypy/go vet)
- [ ] Unit tests pass
- [ ] Lint checks pass
- [ ] No unresolved merge conflict markers (`<<<<<<<`)
- [ ] All expected files present
- [ ] No unexpected files introduced

### Checking for Residual Conflict Markers

After merging, scan for any leftover conflict markers that may have been missed:

```bash
grep -rn '<<<<<<< \|======= \|>>>>>>> ' --include='*.ts' --include='*.py' --include='*.go' .
```

## Recovery from Failed Integration

### Scenario 1: Merge Conflict

```bash
# 1. Check which subtask caused the conflict
cat merge-results.json | jq '.results[] | select(.status == "conflict")'

# 2. Create a fresh merge worktree
git-prepare.sh $ISSUE --suffix merge-retry --base $BASE

# 3. Manually merge the conflicting branch
cd $NEW_WORKTREE
git merge --no-ff $CONFLICTING_BRANCH

# 4. Resolve conflicts
# Edit conflicting files
git add .
git commit

# 5. Continue with remaining branches
# Re-run merge-subtasks.sh
```

### Scenario 2: Type Check Failure

```bash
# 1. Identify failing checks
npx tsc --noEmit 2>&1 | head -50

# 2. Common fixes:
#    - Add missing imports
#    - Fix type mismatches introduced by merge
#    - Update interface definitions

# 3. Re-run type check after fixes
npx tsc --noEmit

# 4. Commit fixes
git add . && git commit -m "fix: resolve type errors from integration"
```

### Scenario 3: Test Failure

```bash
# 1. Run tests with verbose output
npm test -- --verbose 2>&1 | tail -100

# 2. Use dev-validate with --fix for auto-fixable issues
Skill: dev-validate --fix --worktree $MERGE_WORKTREE

# 3. If tests still fail, investigate specific failures
# Check if test assumptions from one subtask conflict with changes from another

# 4. Commit test fixes
git add . && git commit -m "fix: resolve test failures from integration"
```

### Scenario 4: Complete Restart

If the integration is in an unrecoverable state:

```bash
# 1. Remove the merge worktree
git worktree remove $MERGE_WORKTREE --force

# 2. Reset flow.json integration section
flow-update.sh --flow-state $FLOW_STATE integration --field status --value "pending"

# 3. Re-run dev-integrate from scratch
Skill: dev-integrate --flow-state $FLOW_STATE
```

## Execution Steps

Detailed step-by-step commands for the dev-integrate workflow.

### Step 1: Verify Subtask Completion

```bash
$SKILLS_DIR/_lib/scripts/flow-read.sh --flow-state $FLOW_STATE \
  --field '.subtasks[] | select(.status != "completed") | .id'
```

If any subtask is not completed, abort and report which subtasks are pending.

### Step 2: Detect File Change Drift

```bash
$SKILLS_DIR/dev-integrate/scripts/check-drift.sh --flow-state $FLOW_STATE
```

Warn on differences between planned and actual files changed. Do not abort on drift --
it is informational only.

### Step 3: Determine Merge Order

Use topological sort based on `depends_on` fields. Independent subtasks (no dependencies)
are merged first, followed by subtasks that depend on them.

### Step 4: Create Merge Worktree

```bash
$SKILLS_DIR/git-prepare/scripts/git-prepare.sh $ISSUE --suffix merge --base $BASE
```

### Step 4b: Sync .env Files (git-prepareが自動実行しない場合のみ)

> **Note**: git-prepare.sh は内部で sync-env を自動呼び出しする。
> `--env-mode none` で Step 4 を実行した場合や、手動で再同期が必要な場合のみ以下を実行する。

```bash
$SKILLS_DIR/sync-env/scripts/sync-env.sh --worktree $MERGE_WORKTREE --mode $ENV_MODE --force
```

### Step 5: Merge Subtask Branches

```bash
$SKILLS_DIR/dev-integrate/scripts/merge-subtasks.sh \
  --flow-state $FLOW_STATE --worktree $MERGE_WORKTREE
```

### Step 6: Type Check

Detect project type and run appropriate type checker:

```bash
# Detect project type:
if [ -f "tsconfig.json" ]; then npx tsc --noEmit
elif [ -f "setup.py" ] || [ -f "pyproject.toml" ]; then mypy .
elif [ -f "go.mod" ]; then go vet ./...
fi
```

### Step 7: Integration Validation

```
Skill: dev-validate --worktree $MERGE_WORKTREE
```

### Step 8: Update Flow State

```bash
$SKILLS_DIR/_lib/scripts/flow-update.sh --flow-state $FLOW_STATE \
  integration --field status --value "integrated"
$SKILLS_DIR/_lib/scripts/flow-update.sh --flow-state $FLOW_STATE \
  integration --field merge_worktree --value "$MERGE_WORKTREE"
$SKILLS_DIR/_lib/scripts/flow-update.sh --flow-state $FLOW_STATE \
  integration --field type_check --value "passed"
$SKILLS_DIR/_lib/scripts/flow-update.sh --flow-state $FLOW_STATE \
  integration --field validation --value "passed"
```

## Flow.json Integration Section

After a successful integration, flow.json is updated with:

```json
{
  "integration": {
    "status": "integrated",
    "merge_worktree": "/path/to/merge/worktree",
    "type_check": "passed",
    "validation": "passed",
    "merged_at": "2026-02-09T12:00:00Z",
    "merge_order": ["task1", "task3", "task2"]
  }
}
```

On failure:

```json
{
  "integration": {
    "status": "failed",
    "failed_at_task": "task2",
    "error": "unresolvable merge conflict",
    "conflict_files": ["src/auth/handler.ts"]
  }
}
```
