# Scoring Framework

## Common Criteria (All Task Types)

### requirements (1-10)
Issue の受入基準をどの程度満たしているか。

| Score | Description |
|-------|-------------|
| 9-10 | 全受入基準を完全に満たし、エッジケースも考慮済み |
| 7-8 | 主要な受入基準を満たす。軽微な抜けがある程度 |
| 5-6 | 主要機能は動くが、一部の要件が未対応 |
| 3-4 | 基本機能は実装されているが、要件の理解が不十分 |
| 1-2 | 要件の半分以上が未実装、または大きな誤解がある |

### code_quality (1-10)
可読性、SOLID 原則、命名、適切な抽象度。

| Score | Description |
|-------|-------------|
| 9-10 | 既存パターンに完全に従い、読みやすく保守しやすい |
| 7-8 | 全体的に良好。命名や構造に軽微な改善余地 |
| 5-6 | 動作するが、可読性や構造に明確な問題がある |
| 3-4 | 技術的負債が大きい。リファクタが必要 |
| 1-2 | スパゲッティコード。保守不能 |

### edge_cases (1-10)
異常系・境界値の考慮。

| Score | Description |
|-------|-------------|
| 9-10 | 全ての重要なエッジケースが処理されている |
| 7-8 | 主要なエッジケースは処理。稀なケースが一部未対応 |
| 5-6 | 基本的なエラーハンドリングはあるが、不十分 |
| 3-4 | エラーハンドリングが最小限 |
| 1-2 | 異常系がほぼ考慮されていない |

## Type-Specific Criteria

### frontend: UI一貫性 (1-10)
- アクセシビリティ（aria-*, セマンティックHTML, キーボード操作）
- レスポンシブ対応
- 既存UIパターンとの一貫性
- UXパターン（ローディング、エラー状態、空状態）

### api: API設計 (1-10)
- RESTful 規約への準拠
- エラーレスポンス設計（適切な HTTP ステータス、エラーメッセージ）
- バリデーション（入力検証、型チェック）
- 認証/認可の適切な実装

### refactor: 安全性 (1-10)
- 振る舞い保持（既存テストが全て通る）
- テストカバレッジの維持または向上
- public API に破壊的変更がない
- パフォーマンスの劣化がない

### infrastructure: 信頼性 (1-10)
- 冪等性（繰り返し実行しても安全）
- ロールバック可能性
- セキュリティ設定（secrets が露出していない）
- 設定の安全性（デフォルト値の妥当性）

## Score Calculation

```
タイプ別基準がある場合:
  total = (共通3基準の平均 × 0.7) + (タイプ別基準 × 0.3)

generic (タイプ別基準なし):
  total = 共通3基準の平均
```

Threshold: 7.0 (configurable via skill-config.json)

Note: タイプ別基準がある場合はスコアが下がりやすい。これは意図的な設計 —
タイプ固有の品質基準も満たすことを求めるため。

## feedback_level Classification

| Level | Examples |
|-------|---------|
| design | アーキテクチャの問題、API設計の根本的な誤り、データモデルの不整合、責務の分離不足 |
| implementation | バグ、命名の問題、エッジケース不足、コードスタイル、パフォーマンス問題 |
