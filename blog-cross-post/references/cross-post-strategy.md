# Cross-Post Strategy

クロスポスト全体の差別化方針と共通ルール。
プラットフォーム固有のテンプレート・変換ルールは各 publish スキルの `references/content-guide.md` を参照。

## 差別化の原則

**同一内容の複数投稿はSEO効果が分散** → プラットフォームごとに切り口を変える

| Platform | 読者層 | 切り口 | 読後感 |
|----------|--------|--------|--------|
| **Zenn** | モダン技術志向、個人開発者文化 | Deep How + 意見（実装 + 判断 + 持論） | 「判断基準ごと持ち帰れた」 |
| **Qiita** | 幅広い技術者、背景理解重視 | Why（技術選定理由） | 「判断できる」 |

### カテゴリ別の切り口

| Category | Zenn版 | Qiita版 |
|----------|--------|---------|
| tech-tips | 課題共感→実装+判断理由→持論 | 技術選定理由・比較検討中心 |
| lab-reports | 課題共感→検証結果+解釈→持論 | 実験の背景・動機・Why |

**NG: 同一内容**
- 両プラットフォームで構成・切り口が同じ → SEO分散、読者にも価値なし

**OK: 切り口が異なる**
- 同じ技術トピックでも読後感が異なる状態を目指す

## 対象カテゴリ

| Category | Cross-post | 理由 |
|----------|-----------|------|
| tech-tips | ✅ 対象 | エンジニア向け、Zenn/Qiita適性高 |
| lab-reports | ✅ 対象 | 技術実験、エンジニア読者に刺さる |
| solutions | ❌ 対象外 | ビジネス層向け、プラットフォーム不適 |
| case-studies | ❌ 対象外 | ビジネス層向け、プラットフォーム不適 |

## コア版の設計原則

**「誘導記事」ではなく「コア版 + 深掘り導線」**

```
コア版で提供するもの:
├── 問題提起（読者の共感）
├── 解決アプローチの概要
├── 動作するコード例（1-2個）
├── 主要な結論
└── 【ここで記事として完結】

深掘り導線で示すもの:
├── 「元記事ではさらに...」
│   ├── 複数パターンの比較
│   ├── 実運用での知見
│   ├── 失敗談とその対処
│   └── 応用・発展的な内容
└── 具体的に何が読めるかを明示
```

### NGパターン

| NG | 理由 | OK |
|----|------|-----|
| 「続きは公式で」 | 途中で切れてる感 | 「さらに詳しく知りたい方へ」 |
| 概要だけで終わる | 読者不満 | コード例まで含める |
| 公式と同一内容 | 差別化なし | コア vs 詳細の明確な差 |

## 元記事から抽出する情報

| 変数 | 抽出元 | 説明 |
|------|--------|------|
| `{CORE_TOPIC}` | 記事の主要テーマ | コア版で扱う範囲 |
| `{CODE_EXAMPLE}` | 元記事のコード | **動作する完全なコード** |
| `{WHY_THIS_APPROACH}` | 元記事の背景・理由 | なぜこのアプローチを選んだか |
| `{ALTERNATIVES}` | 元記事の比較・検討 | 他の選択肢との比較 |
| `{DEPTH_1}` | 元記事の応用セクション | 公式版のみの内容 |
| `{DEPTH_2}` | 元記事の知見セクション | 公式版のみの内容 |
| `{DEPTH_3}` | 元記事の比較・検証 | 公式版のみの内容 |
| `{ORIGINAL_URL}` | Init結果 | `original_url` |
| `{ARTICLE_TITLE}` | 元記事frontmatter | `title` |

## 深掘り導線の書き方（共通）

### 具体的に書く

**NG: 抽象的**
```markdown
- 詳細な実装
- 応用例
- その他の知見
```

**OK: 具体的**
```markdown
- ISR/SSG/SSRそれぞれの実装パターン
- キャッシュ制御のエッジケース対応
- 100記事規模でのビルド時間ベンチマーク
```

## 共通変換ルール

### MDXコンポーネント変換

| Component | Action |
|-----------|--------|
| `<Mermaid>` | ` ```mermaid` コードブロックに変換（`%%{init}%%` は削除） |
| インタラクティブ | `:::message` で元記事へ誘導 |
| コードブロック | 言語指定を維持 |
| テーブル | そのまま |

### パス変換

| 対象 | Before | After |
|------|--------|-------|
| 画像 | `/blog/image.webp` | `https://www.playpark.co.jp/blog/image.webp` |
| 内部リンク | `/contact` | `https://www.playpark.co.jp/contact` |
| アンカー | `#section` | そのまま |

### 削除する要素

| Element | 理由 |
|---------|------|
| frontmatter `description` | Zenn/Qiita不使用 |
| frontmatter `image` | OGP自動生成 |
| お問い合わせCTA | プラットフォーム固有フッターに置換 |

## SEO共通ルール

### 公開タイミング

1. **元記事を先に公開**（最低24-48時間前）
2. Googleがインデックス完了を確認
3. その後Zenn/Qiitaに投稿

### 見出し構造

```markdown
# タイトル (H1 - 1つのみ)
## セクション1 (H2)
### サブセクション (H3)
## まとめ (H2)
## 元記事・関連情報 (H2)
```

### コンテンツ品質チェックリスト

- [ ] タイトルにメインキーワード含む
- [ ] H2見出しにサブキーワード含む
- [ ] 導入文で記事の価値を明示
- [ ] コード例が動作確認済み
- [ ] 画像に alt テキストあり
- [ ] プラットフォーム固有フォーマット適用済み

### seed情報の活用

| seed file | 抽出情報 |
|-----------|---------|
| export.md | 技術スタック、主要機能 |
| context.md | 課題解決、ユースケース |
| pr-summary.md | 機能追加の変遷、改善点 |

### 分析・改善サイクル

1. 記事公開
2. 2週間後に流入分析（GAリファラー）
3. 効果の高いパターンを特定
4. 次回記事に反映
