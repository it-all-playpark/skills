# Platform-Differentiated Templates

ZennとQiitaで**切り口を変えた**テンプレート。同一内容の重複を避け、各プラットフォームのユーザー特性に最適化。

## プラットフォーム特性

| Platform | 読者層 | 好む切り口 | 記事スタイル |
|----------|--------|-----------|-------------|
| **Zenn** | モダン技術志向、個人開発者文化 | **Deep How + 意見（実装 + 判断 + 持論）** | 課題共感→実装+判断理由→持論、書き手の体験として語る |
| **Qiita** | 幅広い技術者、背景理解重視 | **Why（技術選定理由）** | 解説先行、比較検討、判断材料 |

## 重要ルール

### 1. 対象カテゴリの確認

| Category | 対応 |
|----------|------|
| tech-tips | このテンプレートを使用 |
| lab-reports | このテンプレートを使用 |
| solutions | エラー終了 |
| case-studies | エラー終了 |

### 2. 差別化の原則

**NG: 同一内容**
- ZennとQiitaで構成・切り口が同じ
- SEO効果が分散、読者にも価値なし

**OK: 切り口が異なる**
- Zenn: 「こういう課題ありませんか？」→ 実装 + 判断理由 + 持論
- Qiita: 「なぜこうするか」から始まる
- 同じ技術トピックでも読後感が異なる

### 3. 元記事から抽出する情報

| 変数 | 抽出元 | 説明 |
|------|--------|------|
| `{CORE_TOPIC}` | 記事の主要テーマ | コア版で扱う範囲 |
| `{CODE_EXAMPLE}` | 元記事のコード | **動作する完全なコード** |
| `{WHY_THIS_APPROACH}` | 元記事の背景・理由 | なぜこのアプローチを選んだか |
| `{ALTERNATIVES}` | 元記事の比較・検討 | 他の選択肢との比較 |
| `{DEPTH_1}` | 元記事の応用セクション | 公式版のみの内容 |
| `{DEPTH_2}` | 元記事の知見セクション | 公式版のみの内容 |
| `{DEPTH_3}` | 元記事の比較・検証 | 公式版のみの内容 |
| `{ORIGINAL_URL}` | Init結果 | `original_url` |
| `{ARTICLE_TITLE}` | 元記事frontmatter | `title` |

## Category: tech-tips

### Zenn Template (tech-tips) - Deep How + 意見

**切り口: 課題共感 → 実装 + 判断理由 → 持論・トレードオフ**

**Zennで伸びる記事の特徴:**
- TL;DRで始めない（好奇心を殺す）
- コードだけでなく「なぜこう書くか」を添える
- 他の方法との比較・トレードオフを含める
- 書き手の体験・意見が入っている（個人ブログ文化）
- 企業感を抑え、一人称で語る

```markdown
---
title: "{課題・体験ベースのタイトル}"
emoji: "🛠️"

## Category: lab-reports

### Zenn Template (lab-reports) - 課題共感 + 検証 + 持論

**切り口: 「期待と現実のギャップ」から始まり、検証結果 + 解釈 + 持論で締める**

**Zenn lab-reportsで伸びる記事の特徴:**
- 「試してみた」だけでなく「だからこう使うことにした」まで書く
- 期待と現実のギャップを正直に語る
- 失敗や期待外れも含めてリアルに書く（信頼性が上がる）
- 書き手の解釈・判断を明確にする

```markdown
---
title: "{検証テーマ}を{期間/条件}試して分かった現実"
emoji: "🔬"
type: "tech"
topics: ["{topic1}", "{topic2}", "{topic3}"]
published: true

{読者が期待しがちなこと + 実際にはそう簡単ではないという問題提起 - 150-200字}

{検証の目的と条件を簡潔に - 100-150字}

## 深掘り導線の書き方ガイド

### プラットフォーム別の導線ニュアンス

| Platform | 導線のトーン | 例 |
|----------|-------------|-----|
| Zenn | 控えめに「元記事ではさらに〜」 | 記事末尾に1セクション、宣伝感を出さない |
| Qiita | 「さらに詳しい解説」 | 「実運用での知見」「失敗パターンと対処法」 |

**Zenn導線の注意点:**
- 「さらに深掘りしたい方へ」という見出しは使わない（宣伝感が強い）
- 「元記事」というシンプルな見出しで、フラットにリンクする
- playparkの企業紹介セクションは**削除**（Zennは個人ブログ文化、企業感は嫌われる）

### 具体的に書く

**NG: 抽象的**
```markdown
- 詳細な実装
- 応用例
- その他の知見
```

**OK: 具体的**
```markdown
# Zenn版（How寄り）
- 🔄 ISR/SSG/SSRそれぞれの実装パターン
- ⚠️ キャッシュ制御のエッジケース対応
- 📊 100記事規模でのビルド時間ベンチマーク

# Qiita版（Why寄り）
- 💭 ISR vs SSG vs SSRをどう選ぶか
- ⚠️ 本番運用で遭遇した問題と判断
- 📋 チーム導入時の検討ポイント
```

## 文字数ガイドライン

| セクション | Zenn | Qiita |
|-----------|------|-------|
| TL;DR/学べること | 100-150字 | 100-150字 |
| 背景・課題 | 100-150字 | 200-300字 |
| 本文（コード・解説） | 800-1200字 | 600-1000字 |
| まとめ | 100-150字 | 100-150字 |
| 深掘り導線 | 150-200字 | 150-200字 |
| **合計** | **1400-2200字** | **1200-2000字** |

**Zennはコード多め、Qiitaは解説多め**になる傾向。
